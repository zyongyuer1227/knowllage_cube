# 法律文档知识库 Web 系统开发需求规格（开发版）

## 1. 文档目的与范围

本文档将 `request.md` 的业务需求转换为可直接用于设计、开发、测试、上线的实现规格，作为后续迭代基线。

- 产品形态：Web 应用
- 文档范围：法律法规类文档（国家/地方/行业）
- 用户角色：仅 `管理员`、`游客`
- 核心能力：`Word/TXT/PDF/HTML -> Markdown`、检索、版本、归档、权限与审计

## 1.1 当前部署环境基线（已确认）

- 操作系统：`Ubuntu 24.04`
- Node.js：`v25.8.0`
- PostgreSQL：`16.13`

## 3. 功能需求（Functional Requirements）

> 优先级：P0（首期必须）、P1（首期建议）、P2（后续迭代）

### 3.1 文档管理

**FR-001 文档上传与解析（P0）**
- 支持单个/批量上传：`doc/docx/txt/pdf/html`
- 上传后进入异步解析队列，输出标准化 `Markdown`
- 解析失败可重试，失败原因可见
- 验收标准：
  - 支持批量上传 >= 100 文件/批
  - 成功文件生成可预览 Markdown
  - 失败文件保留错误码与错误信息

**FR-002 元数据管理（P0）**
- 上传或编辑时维护字段：文档类型、层级、领域、关键词、生效日期、失效日期、发布机关、文号、地域、状态（生效/失效/作废）
- 验收标准：
  - 必填字段校验生效
  - 日期字段支持范围校验与空值策略

**FR-003 文档编辑（P1）**
- 在线编辑 Markdown 正文与元数据
- 支持“复杂文档重新上传”替换内容
- 验收标准：
  - 编辑提交后自动生成新版本
  - 保留修改人、时间、说明

**FR-004 分类归档（P0）**
- 按层级/领域/时间自动归档
- 管理员可手动调整归档路径
- 目录层级最大 5 级
- 验收标准：
  - 自动归档规则可配置
  - 超过 5 级阻止保存并提示

**FR-005 版本管理（P0）**
- 每次正文或关键元数据变更产生版本快照
- 支持版本列表、版本对比、回滚
- 验收标准：
  - 可查看任意两版本差异
  - 回滚生成新版本，不覆盖历史

**FR-006 删除/作废/回收站（P0）**
- 删除进入回收站，默认保留 30 天（可配置）
- 支持恢复与永久删除
- 作废：不删除正文，但默认不参与常规检索
- 验收标准：
  - 回收站到期自动清理
  - 作废文档必须记录原因与日期

### 3.2 检索查询

**FR-007 多维检索（P0）**
- 全文检索、标题检索、分类筛选、精准字段检索（法规名/发布机关/文号）
- 验收标准：
  - 支持组合筛选并返回高亮结果

**FR-008 高级检索与排序（P0）**
- 组合查询（关键词 + 领域 + 时间 + 状态）
- 排序：相关性、发布时间、更新时间、发布主体
- 验收标准：
  - 查询条件可保存（仅管理员）
  - 游客无“保存检索条件”能力

**FR-009 搜索联想（P1）**
- 输入时联想法规名称、法律术语、历史热门词
- 验收标准：
  - 联想响应时间 <= 300ms（应用内缓存命中）

**FR-010 结果展示与预览（P0）**
- 结果列表展示核心属性与摘要
- 文档预览基于 Markdown 渲染
- 验收标准：
  - 游客可预览全部可见文档
  - 游客无下载/分享入口

### 3.3 权限与安全

**FR-011 鉴权与会话（P0）**
- 管理员账号密码登录 + 验证码
- 管理员凭据直接读取服务端 `.env`
- 游客免登录访客态
- 验收标准：
  - 管理员接口均需鉴权
  - 游客访问管理接口返回 403

**FR-012 RBAC 权限控制（P0）**
- 仅两角色：管理员/游客
- 菜单、按钮、接口三层一致控制
- 验收标准：
  - 禁止前端隐藏但后端可调用的越权情况

**FR-013 操作审计（P0）**
- 记录管理员关键操作：上传、编辑、删除、回滚、导出、系统配置
- 验收标准：
  - 审计日志可按人/时间/操作类型检索

### 3.4 智能辅助

**FR-014 标签系统（P1）**
- 标签支持手动维护与自动识别（规则/模型均可）
- 游客仅可基于标签筛选

**FR-015 关联推荐（P2）**
- 根据关键词、引用关系、主题相似度推荐相关法规

**FR-016 到期提醒（P1）**
- 针对生效/失效日期配置提前 7/15 天提醒
- 通道：站内消息、邮件

**FR-017 批量操作（P1）**
- 批量标注、批量改属性、批量导出（管理员）

### 3.5 统计分析

**FR-018 统计看板（P1）**
- 文档统计：总量、分层级、分领域、新增、失效
- 检索统计：热词、频次、成功率
- 操作统计：管理员活跃与操作分布

**FR-019 报表导出（P1）**
- 导出 Excel（统计 + 操作数据）

### 3.6 集成能力

**FR-020 外部系统对接（P2）**
- 预留 OA/法务系统同步接口（管理员触发）

## 4. 非功能需求（Non-Functional Requirements）

**NFR-001 性能**
- 简单检索 P95 <= 0.5s
- 全文检索 P95 <= 1s
- 批量检索 P95 <= 3s

**NFR-002 并发与容量**
- 目标同时在线 <= 20（按 20 并发容量设计，建议预留到 30 峰值）
- 初始存储 >= 100GB，支持在线扩容

**NFR-003 可用性与恢复**
- 年可用性 >= 99.9%
- 故障恢复时间（RTO）<= 1 小时
- 数据恢复点目标（RPO）<= 24 小时

**NFR-004 备份**
- 每日自动备份，保留 30 天
- 管理员支持手动全量备份与恢复

**NFR-005 安全**
- 传输加密（HTTPS/TLS1.2+）
- 存储加密（对象存储或磁盘加密）
- 上传文件病毒扫描必开

**NFR-006 兼容性**
- 浏览器：Chrome/Edge/Firefox 最新两个稳定版本
- 平台：Windows/Mac 主流桌面端访问

- `current_version`, `archive_path`, `created_by`, `updated_by`
- `created_at`, `updated_at`

## 9. 测试与验收标准

### 9.1 功能验收
- P0 功能全部通过用例，禁止阻断级缺陷上线
- 游客不可访问任何管理接口与管理数据

### 9.2 性能验收
- 在 20 并发（峰值压测 30）下满足 NFR 响应指标

### 9.3 安全验收
- 权限越权测试通过
- 上传病毒样本拦截有效
- 审计日志完整可追溯

### 9.4 数据验收
- 每日备份任务成功率 100%
- 恢复演练可在 1 小时内完成

## 10. 开发约束与技术原则

- 优先成熟技术栈，避免高维护成本方案
- 解析链路可插拔，便于后续优化 OCR/转换质量
- 所有关键业务操作均事件化记录，便于审计与统计复用
- 对外接口统一版本化（`/api/v1` 建议）

## 11. 技术选型（轻量主流版，非 Python）

### 11.1 推荐主栈（首期）
- 前端：`Vue 3 + TypeScript + Vite + Pinia + Vue Router + Element Plus`
- 后端：`Node.js v25.8.0 + NestJS`
- 数据库：`PostgreSQL 16.13`（同时承担结构化数据与全文检索）
- 检索实现：`PostgreSQL Full-Text Search + pg_trgm`（首期不强依赖 Elasticsearch）
- 文件存储：`MinIO` 或本地/NAS 文件系统（按部署环境选择）
- 文档转换：`LibreOffice headless + Pandoc`（统一输出 Markdown）
- 病毒扫描：`ClamAV`
- 定时任务：`NestJS Schedule`（到期提醒、备份任务）
- 部署：`Ubuntu 24.04 + Nginx + Node.js PM2/Systemd`（单体部署，后续可升级）

### 11.2 架构边界（避免过度设计）
- 首期采用单体架构，不拆微服务
- 首期不引入 `Redis`、`Elasticsearch`、`RabbitMQ`、`Kubernetes`、`Docker Compose`
- 当文档量和并发明显增长时，再按需升级为 `ES + MQ + K8s`

### 11.3 选型理由
- 全栈主流、实现成本低，适配 <=20 并发场景
- Postgres 内建全文能力可覆盖首期检索需求，降低运维复杂度
- 单体部署更快上线，后续保留平滑扩展路径

### 11.4 明确不采用
- 不采用 `Python + Flask` 作为主业务后端框架
- 不采用小众或社区活跃度低的核心技术作为基础依赖

### 11.5 可替代主流方案（如团队偏好）
- 后端可替换为：`Java 21 + Spring Boot 3`（仍建议单体架构）
- 前端可替换为：`React + TypeScript`
- 搜索升级可替换为：`Elasticsearch` 或 `OpenSearch`

## 12. 出界说明

以下内容不在系统责任边界内：
- 法律文本内容真实性、法律效力判断（由管理员负责）
- 自动法律意见输出与法律责任承担
